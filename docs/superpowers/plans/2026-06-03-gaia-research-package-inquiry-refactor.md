# Gaia Research package/inquiry-centric refactor plan

Date: 2026-06-03

## Goal

Refactor `gaia research` so Explore, Focus, Expand, and Assess keep the live research
state in Gaia package source plus `.gaia/inquiry`, while `.gaia/research` remains an
audit/cache layer for trace artifacts, rendered reports, and reproducibility.

The intended contract is:

- `explore` performs broad search scans and records candidate directions as inquiry
  hypotheses/obligations, without pulling papers or writing formal claims.
- `focus` promotes the top 3 accepted LLM-generated focuses into `question(...)`
  statements and makes the primary question the active inquiry focus.
- `expand` performs targeted search around a focus or obligation, still without paper
  pulls.
- `assess` is the first stage allowed to pull/pin deeper evidence; it records review
  scaffolds, unresolved obligations, and only writes `candidate_relation(...)` when the
  analysis supplies concrete claim references.
- `promote` is the narrow bridge from scaffold to formal Gaia claims/relations; it should
  not become a hidden orchestration layer.

This slice deliberately stops before LKM/search evidence source-package
materialization. The next package-native slice should use `gaia pkg add --local
<source-package>` to attach shallow or deep local evidence packages generated by
Explore/Assess, so fast and deep modes share the same package dependency
contract instead of growing artifact-local `items` into long-lived knowledge
entities.

## Milestone 1: research sync layer

Add `gaia.engine.research.sync` with idempotent helpers that can:

- append `question(...)`, `note(...)`, `candidate_relation(...)`, and `materialize(...)`
  statements through the same authored-submodule convention used by `gaia author`;
- update `.gaia/inquiry/state.json` with synthetic hypotheses, obligations, and focus;
- append inquiry tactic-log events for every state mutation;
- return a structured summary for CLI output and `.gaia/research/events.jsonl`.

Success criteria:

- repeated sync calls do not duplicate authored statements or identical inquiry entries;
- authored output lands in `src/<import_name>/authored/__init__.py`;
- root package `__init__.py` is composed through the existing authored re-export helper;
- no `.gaia/research/*` artifact becomes the canonical source of focus/obligation state.

## Milestone 2: Explore and Expand write inquiry state

Wire `gaia research explore` and a first-class `gaia research expand` command to sync
landscape candidate focuses and coverage gaps into inquiry state.

Success criteria:

- `explore --dry-run` remains source- and inquiry-preserving;
- live `explore` writes landscape artifacts plus inquiry hypotheses/obligations;
- `expand` requires exactly one `--focus` or `--obligation`, writes targeted landscape
  artifacts, and never pulls papers;
- event payloads expose `writes_inquiry`, `writes_source`, and sync counts.

## Milestone 3: Focus writes accepted questions

Wire `gaia research focus` so accepted focuses are not just JSON rows.

Success criteria:

- default `focus` writes at most 3 accepted focuses as `question(...)`;
- the first written question becomes the active inquiry focus;
- coverage gaps become inquiry obligations;
- non-accepted focuses remain artifact-local or inquiry hypotheses, not package questions;
- `--artifact-only` preserves the old source/inquiry-preserving behavior.

## Milestone 4: Assess writes review scaffolds

Wire `gaia research assess` so it records meaningful review state without pretending that
candidate analysis is formal knowledge.

Success criteria:

- default `assess` writes candidate obligations and natural-language relation hypotheses
  into inquiry state;
- review text is recorded as a package `note(...)` scaffold;
- `candidate_relation(...)` is written only when an assessment relation carries at least
  two concrete claim references;
- `--artifact-only` remains available for pure trace/report generation;
- event payloads include relation, obligation, note, and candidate-relation counts.

## Milestone 5: Promote remains narrow

Add the minimal `gaia research promote` bridge for explicit scaffold materialization.

Success criteria:

- `promote` requires a scaffold binding and at least one concrete materialized target;
- it writes a `materialize(...)` statement through the authored-submodule path;
- it does not run explore/assess/report implicitly.

## Milestone 6: tests and docs

Update CLI tests and foundations docs to reflect the new default state flow.

Success criteria:

- PR-gate research CLI tests cover default writes and artifact-only opt-outs;
- `uv run pytest tests/cli/test_research.py -q` passes;
- `uv run gaia research --help` shows `expand` and `promote`;
- docs no longer describe Explore/Focus/Assess as artifact-only by default.
