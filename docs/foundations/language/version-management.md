## 1. Design Principles

1. **Package semver, auto-derived.** The system determines the version bump (patch/minor/major) from the manifest diff. Authors declare intent; the system verifies.

2. **Knowledge has no independent version.** Content hash tracks changes. Package version fully determines every knowledge unit's content.

3. **Lock, don't range.** Dependencies are pinned to exact versions, like citing a specific paper. No range constraints. Updates are explicit.

4. **No administrative lifecycle.** No `superseded` or `retracted` status. Conclusions are challenged by publishing counter-evidence; BP handles the rest.

## 2. Version Model

```
Package:     name @ semver (X.Y.Z)
Knowledge:   name + content_hash (no independent version number)
Manifest:    {knowledge_name → content_hash, ...}
```

- Package semver is auto-derived from the manifest diff against the previous published version.
- Knowledge content hash enables change detection, deduplication, and integrity verification.
- Package version + knowledge name fully identifies the content.

## 3. Semver Semantics

| Level | Trigger | Examples |
|-------|---------|----------|
| **Patch** (x.y.Z) | No semantic change | Typo fix, wording clarification, minor prior adjustment |
| **Minor** (x.Y.z) | Semantic change or expansion, no downstream breakage | New knowledge/chain/module, new export, internal restructuring |
| **Major** (X.y.z) | Breaking interface change | Exported knowledge content substantively modified, export removed |

### Classification criteria

The key question: **if a downstream package references your exported knowledge, does its reasoning still hold?**

- Yes, unchanged → Patch
- Yes, but the package grew / internals changed → Minor
- Potentially not → Major

### Detailed case mapping

| Case | Change | Level |
|------|--------|-------|
| Intermediate typo fix | Content (no semantic change) | Patch |
| Intermediate semantic change, may affect export belief | Content + belief | Minor |
| Export typo fix | Content (no semantic change) | Patch |
| Export substantive content change | Content (semantic change) | Major |
| Prior adjustment | Metadata | Patch |
| New knowledge + chain, exports unchanged | Structure | Minor |
| New export added (non-breaking expansion) | Interface | Minor |
| Export removed (breaking) | Interface | Major |

## 4. Modification Principles

### Review routing

| | Patch | Minor | Major |
|--|-------|-------|-------|
| **Review** | None — auto-verify | Lightweight (single engine) | Full peer review |
| **Trigger BP?** | No | Yes | Yes |
| **BP scope** | — | Local propagation | Wide propagation |
| **Downstream notification?** | No | No | Yes |
| **Downstream action?** | None | None | Confirm references or pin old version |

### BP update strategy

Full global recomputation is infeasible at scale. Incremental propagation:

- **Patch:** No propagation. Semantic content unchanged. Minor prior adjustments batched / lazily updated.
- **Minor:** Local propagation. From changed nodes, propagate along factor graph until convergence (delta < ε). Typically attenuates within a few hops.
- **Major:** Wide propagation. From changed export nodes, propagate to all downstream referencing packages. Each triggers local BP recomputation. Layer-by-layer until global convergence.

## 5. Submission Commands

```bash
gaia publish              # Initial publication → v1.0.0
gaia publish patch        # No semantic change
gaia publish minor        # Semantic change / expansion
gaia publish major        # Breaking interface change
```

### Verification rules

- **No under-reporting:** Declared level must be ≥ detected level. `publish patch` with structural changes → rejected.
- **Over-reporting allowed:** `publish major` for a minor change → accepted, routes to stricter review.

### Build-time preview

`gaia build` shows the diff and projected version:

```
Changes detected (vs v1.2.0):
  modified:  reasoning.inclined_obs (content, hash changed)
  added:     reasoning.new_evidence
  structure: 1 new chain
  export:    unchanged

Projected: minor (v1.3.0)
```

## 6. Dependency Management

### Pinning model

Dependencies are pinned to exact versions. No range constraints (`^`, `>=`, `~`).

Analogy: citing a paper — you cite "Newton 1687", not "Newton >=1687". Your reasoning was reviewed against specific content.

```yaml
# package.yaml
dependencies:
  - package: newton_principia

# gaia.lock (generated by gaia build)
newton_principia: 2.3.1
```

### Resolution workflow

| Command | Behavior |
|---------|----------|
| `gaia build` (no lock) | Resolve to latest approved versions, generate `gaia.lock` |
| `gaia build` (with lock) | Use locked versions, deterministic compilation |
| `gaia update <package>` | Re-resolve to latest, update lock. Counts as revision — needs review |

No constraint solver needed. Lock file is a simple `{name: version}` mapping.

## 7. No Supersedes / Retraction

Knowledge lifecycle is managed entirely through BP, not administrative status.

To challenge existing conclusions, publish a new package with counter-evidence:

```
new_package references old_package.conclusion_X
  → provides counter-evidence via reasoning chain
  → BP recomputes
  → old conclusion's belief decreases naturally
```

Package has exactly one status: **approved**. Everything else — declining relevance, superseded conclusions, refuted claims — is expressed through the knowledge graph and resolved by BP.

| Academic concept | Gaia equivalent |
|-----------------|-----------------|
| Retraction | New package with counter-evidence |
| Superseding paper | New package with better reasoning |
| Citation decline | Belief decrease via BP |

## 8. Version Coexistence

When a Major version is published, old and new may coexist (downstream packages may pin the old version).

| Level | Coexist? | Graph connection? |
|-------|----------|-------------------|
| Patch | No (old retires) | Not needed |
| Minor | No (interface unchanged) | Not needed |
| Major | Yes | Auto-created factor between old/new exports, participates in BP |

Major version connections are temporary. When the last downstream package updates its dependency, the old version retires from the active graph to archive.

## 9. Open Questions

1. **Patch auto-verification** — how to verify "no semantic change"? Structural diff is necessary but is it sufficient?
2. **Major version notification** — push vs. lazy detection on next `gaia build`?
3. **Concurrent publishes** — conflict detection for simultaneous revisions?
4. **BP batch scheduling** — incremental BP on every publish or batched?
