# 审核与策展

> **Status:** Current canonical

本文档描述 Review Server 和 Curation Server 的业务逻辑——新证据如何被审核激活，跨包关系如何被自动发现和维护。

## Review Server

### 为什么需要 Review

新注册的推理链默认不参与推理（见 [registry-operations.md](registry-operations.md)）。审核的目的是：

1. **判定新推理和已有推理的关系** — 独立证据、重复论证、还是细化？
2. **赋予推理参数** — 这条推理有多强？
3. **激活推理链** — 审核通过后，推理链开始影响全局可信度。

### 谁来审核

Reviewer 可以是人类专家或 AI agent。Review Server 为 reviewer 提供工具支持：

- 展示新推理链的前提、结论、推理过程
- 展示 official repo 中已有的相关推理链，方便对比
- 建议可能的关系判定（但最终决定权在 reviewer）
- 自动验证 review 提交的格式合规性

### 审核什么

Reviewer 面对的核心问题是：**这条新推理和 official repo 中已有的推理是什么关系？**

当多条推理链指向同一个结论时，它们之间的关系决定了证据如何汇聚：

| 关系 | 含义 | 判定依据 | 对推理结果的影响 |
|------|------|---------|----------------|
| **独立证据** | 不同实验/方法/数据得出的独立论证 | 推理过程本身是独立的，不是前提是否相同 | 独立证据汇聚，增强结论的可信度 |
| **重复论证** | 同一论证的不同表述 | 推理过程本质相同，只是措辞不同 | 不额外贡献——避免 double counting |
| **细化** | 对已有粗粒度推理提供了更精细的内部结构 | 展开了已有推理的步骤，但整体论证相同 | 提供更精细的分析视图，折叠视图不变 |

**为什么只有 reviewer 能判断：** 独立性取决于推理过程本身——两条推理链的前提可能完全相同（都引用了同一组实验数据），但推理方法不同（一个用统计分析，一个用物理模型），这是独立证据。反过来，前提不同但推理方法相同（用同一个模型分析不同但相关的数据集），可能不算独立。这种判断需要理解推理的内容，无法从结构自动推断。

### 审核的具体流程

```
1. Reviewer 查看待审队列中的新推理链
     ↓
2. Review Server 展示：
   - 新推理链的前提、结论、推理过程（steps）
   - 该结论在 official repo 中已有的其他推理链
   - 前提重叠度分析（辅助信息，不是判定依据）
     ↓
3. Reviewer 做出判定：
   a. 独立证据 → 赋予推理参数（这条推理有多强？条件概率多少？）
   b. 重复论证 → 标记为重复，不赋参数，保留记录
   c. 细化 → 标记为细化，链接到被细化的推理链
     ↓
4. Reviewer 提交 review PR 到 official repo
     ↓
5. CI 自动验证（review.yml）：
   - 引用的命题在 official repo 中存在
   - 参数值在合理范围内（遵守 Cromwell's rule：不允许 0 和 1）
   - 审核来源信息完整
     ↓
6. 合并 → 触发增量推理（只计算受影响的局部子图）
```

### 审核后的效果

**独立证据激活后：**
```
之前：结论 X 只有推理链 A 支持，X 的可信度 = 0.80
审核：推理链 B 被确认为独立证据
之后：结论 X 有两条独立推理链支持，X 的可信度 = 0.92
```

独立证据汇聚是可信度提升的核心机制——越多独立的论证指向同一结论，该结论越可信。

**重复论证标记后：**
```
推理链 C 被标记为推理链 A 的重复
→ 不赋参数，不参与推理
→ 保留在记录中，但不额外贡献可信度
```

**细化标记后：**
```
推理链 D 细化了推理链 A 的内部结构
→ 折叠视图：和之前一样（A 作为一个整体）
→ 展开视图：可以看到 A 内部的细化步骤（更精细的分析）
```

## Curation Server

### 为什么需要自动策展

人类 reviewer 处理的是单条推理链的审核。但有些跨包关系需要在全局视角下才能发现：

1. **语义重复命题**——两个命题措辞不同但含义相同，注册时 embedding 匹配漏掉了
2. **跨包连接**——两个包讨论相关话题但互相不知道对方的存在
3. **矛盾检测**——两个命题互相矛盾，但各自的作者不知道对方
4. **知识图健康**——孤岛节点、断裂的推理链、陈旧的引用

这些需要一个能看到全局知识网络的自动化服务。

### Curation Server 的运作方式

**核心原则：Curation Server 没有特殊权限。** 它和普通贡献者一样，通过 PR 与 official repo 交互。它的"自动发现"以**提交新包或 PR 的方式**贡献给 official repo，而不是直接修改数据。

### 策展操作一：发现语义重复命题

```
运行方式：
  Curation Server 定期扫描 official repo 中所有命题
  使用 embedding 模型计算语义相似度
  发现命题 A ≈ 命题 B（措辞不同但语义相同）
    ↓
提交 merge PR 到 official repo：
  - 提议将 B 合并到 A（或反过来）
  - 附带相似度分数和理由
    ↓
Reviewer 审核 merge PR：
  - 确认两个命题确实是同一个
  - 或拒绝（措辞相似但含义不同）
    ↓
合并后的处理：
  a. B 的所有引用重定向到 A
  b. B 标记为已合并（保留审计记录）
  c. 受影响的推理链暂停参数（回退到安全状态）
  d. 级联 re-review：重新评估受影响推理链的独立性
  e. 增量推理
```

**为什么暂停参数：** 合并后，原本指向 A 和 B 两个独立命题的推理链现在都指向 A。如果之前它们被判定为"独立证据"，现在需要重新评估——因为它们可能实际上是在讨论同一个命题，不应该 double counting。暂停参数确保在 re-review 完成前回退到保守状态（可能少算证据，但不会多算）。

### 策展操作二：发现跨包连接

```
运行方式：
  Curation Server 分析知识图谱中的命题和推理链
  发现 Package X 的结论和 Package Y 的前提高度相关
  但 X 和 Y 之间没有依赖关系
    ↓
Curation Server 创建一个 curation 包：
  - 声明 X 的结论和 Y 的前提之间的关系
  - 可能是等价关系、支持关系或矛盾关系
    ↓
以标准注册流程提交到 official repo
  ↓
进入待审队列 → Reviewer 审核 → 正常流程
```

**关键设计：Curation Server 以提交新包的方式贡献**，而不是直接修改已有包。这保持了数据的不可变性和审计性。

### 策展操作三：矛盾检测

```
运行方式：
  Curation Server 分析命题之间的语义关系
  发现命题 P 和命题 Q 互相矛盾
  （如 "Tc = 92K" vs "Tc = 89K"）
    ↓
Curation Server 创建一个 curation 包：
  - 声明 P 和 Q 的矛盾关系
  - 附带检测依据
    ↓
以标准注册流程提交到 official repo
  ↓
Reviewer 确认矛盾 → 推理引擎自动压低双方的可信度
```

**矛盾是结构性事实，不是判断。** 一旦确认两个命题矛盾，推理引擎保证它们不会同时具有高可信度——这是逻辑一致性的自动维护。

### 策展操作四：知识图健康维护

```
Curation Server 定期检查：
  - 孤岛命题（没有任何推理链连接）→ 标记，提醒社区
  - 断裂的依赖（引用的包不再可用）→ 标记，提醒维护者
  - 陈旧的引用（依赖的包有重大更新但下游未跟进）→ 提醒下游
  - 审核积压（待审队列过长）→ 提醒 reviewer 社区
```

这些是信息性的，不修改数据。

## Review Server 和 Curation Server 的关系

| | Review Server | Curation Server |
|---|---|---|
| **触发** | 新推理链进入待审队列 | 定期扫描 + 事件触发 |
| **视角** | 单条推理链 vs 已有推理 | 全局知识网络 |
| **判断主体** | Reviewer（人类/agent），Server 辅助 | Server 自动发现，Reviewer 确认 |
| **产出** | review PR（参数赋值） | curation PR / curation 包 |
| **与 official repo 的交互** | PR | PR / 标准注册流程 |
| **权限** | 无特权，和普通贡献者一样 | 无特权，和普通贡献者一样 |

两者互补：Review Server 处理"已知的新证据"（有人主动提交了），Curation Server 发现"未知的关系"（没人主动建立，但全局视角下存在）。
